## Problem Statement

TeslaCam 的錄影功能很好用，但重播與整理往往又慢又麻煩。Tesla 車主在以下場景中需要快速找到關鍵片段：

1. **事故/證據**：出了車禍，需要快速找到關鍵片段、截取證據、匯出給保險公司或警察
2. **哨兵模式管理**：Sentry 事件太多，需要快速過濾、清理、保留重要的
3. **日常回看**：想回看今天開車的片段，看看風景或有趣的事

現有工具（TeslaCamPlayer、TeslaCam Viewer II、Sentry Studio 等）都只解決了「播放」這一步，沒有人完整解決「找到 → 備份 → 匯出帶水印影片 → 清理」的工作流。而且大多數是 Windows 或瀏覽器方案，macOS 原生支援很弱。

**核心洞察**：Camzy (iOS) 做得最接近，但手機處理 29GB+ 影像檔太吃資源。桌面工具才是正確的平台。

## What Makes This Cool

1. **元資料優先架構**：不是檔案瀏覽器，而是事件引擎。利用 Tesla MP4 內嵌的 SEI 元資料（車速、方向盤角度、自駕狀態）建立事件索引，讓數據成為主角
2. **六鏡頭同步播放 + 遙測覆蓋**：前/後/左後/右後/左柱/右柱全部同步，前鏡頭疊加即時車速、方向盤角度
3. **地圖模式**：在地圖上一眼找到所有事件的位置，空間資訊一目了然
4. **一鍵工作流**：從「找到片段」到「匯出帶時間水印的證據影片」只需幾次點擊
5. **跨平台**：macOS + Windows + Linux，所有 Tesla 車主都能用
6. **多語系**：初版以正體中文 + 英文為主，使用 i18n 框架（如 react-i18next）從 P0 開始架設，方便後續擴展

## Constraints

- **Tauri (Rust + WebView)** 跨平台框架，React/TypeScript 前端
- 所有影片處理在本地完成，不上傳任何資料（隱私優先）
- MP4 檔案體積大（單鏡頭 ~40MB/分鐘，六鏡頭 ~280MB/分鐘），需考慮記憶體和效能
- SEI 元資料解析需要研究 Tesla 的格式（參考 tesla/dashcam 官方工具）
- WebView 同時播放 6 路 HD 視訊可能有效能瓶頸，需要最佳化策略

## Premises

1. ✅ 核心痛點不是「看影片」，而是「從大量檔案中快速找到關鍵片段，並完成備份/匯出/清理的完整工作流」
2. ✅ 跨平台（Tauri）是正確選擇——要分享給其他 Tesla 車主，不能只限 macOS
3. ✅ Tesla MP4 內嵌的 SEI 元資料是大寶藏——車速、方向盤角度、GPS 座標讓地圖模式和遙測覆蓋不需要額外硬體

## Approaches Considered

### Approach A: Tauri 跨平台應用（已選）
- Tauri (Rust 後端 + WebView) + React/TypeScript 前端
- HTML5 Video 播放六鏡頭，Rust 處理檔案操作和 ffmpeg 匯出
- MapLibre GL JS 做地圖，SQLite 儲存 SEI 元資料索引
- 優點：跨平台、現代技術棧、快速疊代
- 缺點：WebView 同時播 6 路 HD 視訊可能卡頓
- 工作量：M（P0 功能：人工 ~2 週 / CC ~4 小時；P0+P1 全部：人工 ~1 個月 / CC ~8 小時）

### Approach B: macOS 原生 SwiftUI 應用
- SwiftUI + AVFoundation + Metal 硬體加速
- 最佳效能但僅 macOS
- 工作量：L（P0 功能：人工 ~3 週 / CC ~6 小時）

### 設計原則：事件引擎 + 元資料優先
已併入方案 A。核心理念：不以資料夾結構組織影片，而是掃描所有 MP4 的 SEI 元資料建立「事件」資料庫。UI 以時間軸 + 地圖為中心，而非檔案瀏覽器。

## Recommended Approach

**Tauri 跨平台 + 事件引擎架構**

### 資料模型

```sql
-- 事件（一組同時間的影片片段）
CREATE TABLE events (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  type        TEXT NOT NULL,        -- 'sentry' | 'saved' | 'recent'
  timestamp   TEXT NOT NULL,        -- ISO 8601, e.g. '2026-03-22T19:09:00'
  duration_s  INTEGER,              -- 事件時長（秒）
  gps_lat     REAL,                 -- GPS 緯度（來自 SEI，可為 NULL）
  gps_lon     REAL,                 -- GPS 經度（來自 SEI，可為 NULL）
  avg_speed   REAL,                 -- 平均車速 km/h（來自 SEI）
  max_speed   REAL,                 -- 最高車速 km/h
  source_dir  TEXT NOT NULL,        -- 原始資料夾路徑
  backed_up   INTEGER DEFAULT 0,    -- 是否已備份
  notes       TEXT                  -- 使用者備註
);

-- 影片片段（每個鏡頭角度一個）
CREATE TABLE clips (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  event_id    INTEGER NOT NULL REFERENCES events(id) ON DELETE CASCADE,
  camera      TEXT NOT NULL,        -- 'front' | 'back' | 'left_repeater' | 'right_repeater' | 'left_pillar' | 'right_pillar'
  file_path   TEXT NOT NULL,        -- 完整檔案路徑
  file_size   INTEGER,              -- 位元組
  duration_s  REAL,                 -- 片段時長
  has_sei     INTEGER DEFAULT 0     -- 是否成功解析 SEI
);

-- SEI 遙測資料快取（每秒一筆或每個 keyframe 一筆）
CREATE TABLE telemetry (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  clip_id     INTEGER NOT NULL REFERENCES clips(id) ON DELETE CASCADE,
  offset_ms   INTEGER NOT NULL,     -- 距片段開始的毫秒數
  speed       REAL,                 -- km/h
  steering    REAL,                 -- 方向盤角度（度）
  gps_lat     REAL,
  gps_lon     REAL,
  drive_state TEXT                  -- 'P' | 'D' | 'R' | 'N'
);

CREATE INDEX idx_events_timestamp ON events(timestamp);
CREATE INDEX idx_events_type ON events(type);
CREATE INDEX idx_clips_event ON clips(event_id);
CREATE INDEX idx_telemetry_clip ON telemetry(clip_id);
```

### 功能優先級

#### P0 — 核心功能（MVP）
1. **USB 磁碟掃描 + 事件索引**
   - 選擇 TeslaCam 根目錄
   - 掃描 RecentClips / SavedClips / SentryClips（跳過 EncryptedClips，顯示提示「X 個加密片段已略過」）
   - 解析檔名取得時間和鏡頭角度
   - 按事件分組：一個「事件」定義為共享相同時間戳前綴的所有鏡頭檔案（例如 `2026-03-22_19-09-00` 開頭的所有 MP4）。一個 Sentry 事件資料夾可能包含多個連續事件，以 `source_dir` 分組。舊款車（HW2.5）可能只有 4 鏡頭（無 pillar），動態適應佈局
   - 建立 SQLite 索引
   - 處理異常：跳過損壞/不完整的 MP4 檔案，記錄到 error log

2. **多鏡頭同步播放**
   - 前鏡頭放大居中（2:1 比例）
   - 其他可用鏡頭環繞（動態佈局：自動適應 4~6 個鏡頭，舊款車可能只有 4 個——無 left_pillar/right_pillar）
   - 同步策略：以前鏡頭為主時鐘，其他鏡頭在每次 `timeupdate` 事件時 seek 對齊，允許 ≤100ms 偏移。若 WebView 支援 `requestVideoFrameCallback` 則使用更精確的逐幀同步
   - 同步播放/暫停/快進/快退
   - 播放速度控制（0.5x ~ 4x）
   - 點擊任何鏡頭可切換為主鏡頭

3. **事件列表 + 快速跳轉**
   - 依類型分組（哨兵/手動保存/行車紀錄）
   - 點擊事件直接跳到該時間點
   - 時間軸上標記所有事件位置

4. **備份 + 刪除**
   - 選擇事件 → 備份到本機指定資料夾
   - 批次選擇 → 一鍵備份多個事件
   - 刪除已備份的事件以釋放 USB 空間

#### P1 — 重要功能
5. **匯出帶水印影片**
   - 選擇要匯出的鏡頭（單鏡頭或多鏡頭合併）
   - 自動加上時間水印（日期 + 時間）
   - ffmpeg 背景處理，顯示進度
   - 輸出格式：MP4 (H.264)

6. **SEI 元資料解析**（依賴 SEI spike 結果，見 Open Questions #1）
   - 提取車速、方向盤角度、GPS 座標
   - 前鏡頭覆蓋顯示即時遙測數據
   - 用速度變化自動標記「急煞車」「急加速」片段
   - **Fallback**：若 SEI spike 失敗（格式無法解析或資料不可靠），GPS/遙測相關功能（#6, #7, P2 #8）全部移到 P2。P1 僅保留水印匯出和批次操作

7. **地圖模式**（Gate：僅在 SEI spike 確認 GPS 資料可靠時進入 P1，否則移至 P2）
   - MapLibre GL JS + OpenStreetMap raster tiles（免費、無 API key）
   - 離線時地圖面板顯示「無網路連線，地圖不可用」，其他功能不受影響
   - 如 SEI 中無 GPS 資料，事件標記為「無位置」，不在地圖上顯示
   - 點擊地圖標記跳到該事件
   - 行車軌跡線顯示

#### P2 — 進階功能（夢想版本，MVP 不預留架構鉤子）
8. **AI 事件自動標記**（利用速度/方向盤資料的規則引擎先做，真正的 AI 後做）——不影響 MVP 資料模型
9. **多車管理**（多個 TeslaCam 根目錄）——如需支援，未來在 events 表加 `vehicle_id` 欄位即可，MVP 不需要
10. **匯出報告**（事件摘要 PDF，給保險公司用）——獨立功能，不影響核心架構

### 效能最佳化策略

| 問題 | 策略 |
|------|------|
| WebView 播 6 路 HD 視訊 | 策略：僅主鏡頭用 `<video>` 播放原始畫質；其他鏡頭用 ffmpeg 在首次掃描時生成低解析度代理檔（360p），或用靜態縮圖 + 點擊切換。需原型驗證最佳方案 |
| 首次掃描 29GB+ 耗時 | 背景掃描 + 進度條，掃描結果快取到 SQLite |
| SEI 解析耗時 | Rust 多執行緒平行解析，結果快取 |
| ffmpeg 匯出耗時 | 背景處理 + 通知完成 |
| 記憶體佔用 | 只載入當前播放片段，預載前後各一個片段 |

## 錯誤處理與邊界情況

| 情境 | 處理方式 |
|------|----------|
| 損壞/不完整的 MP4 檔案 | 跳過該檔案，在事件列表中標記「部分鏡頭遺失」，記錄 error log |
| 掃描時 USB 被拔除 | 中斷掃描，保留已完成的索引，提示使用者重新連接 |
| 備份/刪除時 USB 斷開 | 中斷操作，已完成的備份保留，未完成的標記為失敗 |
| ffmpeg 匯出失敗 | 顯示錯誤訊息 + stderr 內容，保留原始檔案不受影響 |
| 磁碟空間不足（備份目標） | 備份前檢查目標磁碟空間，不足時提示 |
| 事件缺少部分鏡頭（舊款車 4 鏡頭） | 動態佈局，僅顯示可用鏡頭，不顯示空白格 |
| SEI 元資料不存在 | 地圖模式顯示「無位置資料」，遙測覆蓋隱藏，功能正常退化 |
| EncryptedClips | MVP 跳過，顯示提示「X 個加密片段已略過」 |
| GPS 資料不可用 | 地圖面板顯示「此事件無 GPS 資料」，事件仍可正常瀏覽/播放/匯出 |

## Open Questions

1. **SEI 元資料格式**：Tesla 的 SEI NAL unit 使用 H.264 unregistered user data SEI (UUID-based)。`teslamotors/dashcam` 的 JavaScript 實作可做參考，需要用 Rust 重寫解析器。已知欄位包含：vehicle speed, steering angle, GPS coordinates, drive state。不同韌體版本的格式可能有差異——需要用範例影片做跨版本測試。**Next Step：用專案目錄的範例影片做一次 SEI 解析 spike（限時 2 小時），結果決定 P1 #6 的可行性。**
2. **GPS 座標可用性**：並非所有 Tesla 韌體版本都在影片中嵌入 GPS。如果不可用，地圖模式會正常退化（見錯誤處理表）。
3. **加密影片 (EncryptedClips)**：MVP 明確排除。
4. **發佈管道**：初期用 GitHub Releases + Tauri updater plugin。注意：macOS 需要 notarization，ffmpeg sidecar 需與主應用一起 code sign，否則 Gatekeeper 會阻擋。Windows 需要 code signing certificate 避免 SmartScreen 警告。
5. **效能基準**：需要原型驗證——建議先做一個只有 6 個 `<video>` 元素的最小 Tauri 原型，用範例影片測試。

## Success Criteria

- [ ] 首次掃描 128GB USB 磁碟（~50GB TeslaCam 資料，~10,800 個檔案）< 15 秒；後續掃描使用快取索引 < 3 秒
- [ ] 六鏡頭同步播放流暢，無明顯延遲或不同步
- [ ] 從「插入 USB」到「找到目標片段」的時間 < 30 秒
- [ ] 匯出帶水印影片的操作 < 3 次點擊
- [ ] 批次備份 + 刪除一組事件 < 5 次點擊
- [ ] 應用啟動時間 < 3 秒
- [ ] macOS 和 Windows 都能正常運行

## What I noticed about how you think

- 你在被問到核心情境時說「都很重要」——這說明你是從完整使用者體驗出發，而不是只想解決一個小痛點。這對產品設計很好，但在開發時要小心 scope creep
- 你選擇桌面而非手機的理由很務實：「都是影像檔，使用手機較費資源」。你不是追逐趨勢，而是從物理約束出發做技術決策
- 你選 Tauri 跨平台而非 SwiftUI 原生——即使性能稍差，也想讓更多車主能用。你在意分享和社群影響力
- 你的 10x 願景（AI 標記 + GPS 疊加 + 多車管理）很有野心，但你沒有執著於一次做完——你能區分「現在要做的」和「未來可以做的」
